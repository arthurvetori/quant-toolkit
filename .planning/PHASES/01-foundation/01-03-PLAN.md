Build comprehensive unit tests validating handle store thread-safety under high concurrency. These tests MUST pass before any Excel integration; they prove the foundation is sound.

**Test setup (ConcurrencyTestHelpers.cs):**
- Helper: `CreateConcurrentWorkload(int taskCount, int operationsPerTask, Action&lt;int&gt; workload)`
  * Spin up N tasks, each performing M operations
  * Wait for all to complete using Task.WaitAll
  * Capture any exceptions from worker threads
- Helper: `VerifyHandleStoreState(int expectedHandles, int expectedTotalRefCount)`
  * Inspect HandleStore.Count and sum of all RefCounts
  * Assert no data loss or corruption

**Test Class: HandleStoreConcurrencyTests**

1. **Test: ConcurrentCreation_NoCollisions**
   - Scenario: 8 threads, each creates 100 handles (total 800)
   - Validation: All 800 have unique IDs, count == 800
   - Expected: PASS (Guid collisions vanishingly rare)
   - Implementation:
     ```csharp
     [Fact]
     public void ConcurrentCreation_NoCollisions() {
         var store = HandleStore.Instance;
         store.Clear();
         var createdIds = new ConcurrentBag&lt;string&gt;();
         
         CreateConcurrentWorkload(8, 100, _ => {
             var handle = store.CreateHandle(new object());
             createdIds.Add(handle);
         });
         
         Assert.Equal(800, createdIds.Count);
         Assert.Equal(800, createdIds.Distinct().Count()); // No duplicates
     }
     ```

2. **Test: ConcurrentRetrievalAndRefCount**
   - Scenario: Create 10 handles, then 8 threads each read each handle 50 times
   - Validation: All reads succeed; final RefCount = 10 * (1 + 8*50) = 4010 per handle
   - Expected: PASS
   - Implementation pattern: CreateHandle, then Task.Run with Parallel.For(0, 50) doing GetHandle
   - Verify RefCount after all tasks complete

3. **Test: ConcurrentReleaseIntegrity**
   - Scenario: Create 100 handles, 8 threads each release each handle
   - Validation: After all releases, handles are cleaned up (count == 0)
   - Expected: PASS (RefCount atomic decrement works)
   - Implementation: CreateHandle (RefCount=1), then 8 threads ReleaseHandle concurrently, verify handle removed when count reaches 0

4. **Test: InterleavedCreateReadRelease**
   - Scenario: 8 threads randomly create, read, release handles (complex workload)
   - Validation: No deadlocks, no RefCount corruption, no lost data
   - Expected: PASS
   - Implementation: Loop 100x: ~70% reads, ~20% creates, ~10% releases; verify consistency at end

5. **Test: RapidCycling_HighThroughput**
   - Scenario: 8 threads, each: create → read 10x → release (1000 cycles per thread)
   - Validation: 8000 total operations complete without exception; store is clean at end
   - Expected: PASS (stress test the GC and RefCount)
   - Implementation: time the operation; assert completes in &lt; 5 seconds (sanity check performance)

6. **Test: ConcurrentExceptionHandling**
   - Scenario: Mixed valid operations + exception-inducing operations (e.g., GetHandle with invalid ID)
   - Validation: Exception in one thread doesn't corrupt store for others
   - Expected: PASS
   - Implementation: Some threads do valid ops, others try to GetHandle("invalid"); verify store state is consistent after all complete

**Key validation criteria:**
- All tests use xUnit [Fact] or [Theory]
- Tests are deterministic (no flaky timing dependencies)
- Tests complete in &lt; 60 seconds total (fast feedback)
- No deadlocks observed (test harness times out if stuck)
- No exceptions in worker threads (all caught and reported)

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Concurrent Excel threads → Handle Store | Multi-threaded access to RefCount and dictionary |
| User input → UDF parameter validation | Excel cells can contain any formula output (arrays, errors, nulls) |
| Excel → File system (logging) | Logging writes could fail if disk full or permissions denied |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | RefCount under concurrent access | Mitigate | Concurrency test (Task 6) validates Interlocked operations do not corrupt state. Test creates 1000+ concurrent operations; if any RefCount is wrong, test fails. |
| T-03-02 | Information Disclosure | Logs contain user calculations | Accept | Logs are local files; user controls access. No PII encoded. Phase 1 scope. |
| T-03-03 | Denial of Service | Log file exhaustion | Accept | No log rotation limits in Phase 1 spike. Production (Phase 2+) can add rotation policy. |
| T-03-04 | Availability | Excel hangs during unload | Mitigate | Manual test (checkpoint 7) includes unload verification. If hang detected, investigate handle cleanup logic. |

## Mitigation Details

**T-03-01 (RefCount corruption under stress):**
- Validation: Task 6 test suite runs 1000+ operations; validates RefCount consistency
- Test cases: concurrent creation, retrieval, release; interleaved patterns
- Expected: All tests pass, proving no RefCount corruption

**T-03-04 (Excel unload hang):**
- Manual validation in checkpoint: user unloads add-in; Excel must remain responsive
- If hang occurs: likely deadlock in handle cleanup; investigate HandleStore.ReleaseHandle logic
- Mitigation: Add timeout to cleanup operations (future phase if needed)

</threat_model>
